# Fable 5向け Spright UI・ビジュアルレビュー依頼

以下のAndroidアプリ「Spright」について、展示会で来場者が触ることを前提に、UI、ビジュアルデザイン、3Dモデル、モーション、発光表現、アプリアイコンをレビューしてください。

今回はコード修正ではなくレビューを依頼します。一般的な称賛や抽象論だけで終わらず、問題の位置、理由、優先度、具体的な修正案を示してください。良い部分についても、なぜ維持すべきかを説明してください。

## アプリの目的

Sprightは、BLE1507と通信して実物のペンライトを制御するAndroidアプリです。

- 日本語音声を認識する
- 既知色はルール、曖昧・相対・抽象的な表現は端末内のローカルLLMで色へ変換する
- RGBをBLE経由でPWM命令として実物へ送る
- BLE経由で60 HzのIMUを取得する
- roll / pitch / 相対yawを3Dペンライトへ反映する

主な展示端末はXperia 1 VIIで、縦画面・全画面・常時点灯で使用します。

## 展示で伝えたい印象

- 夜空の中に一本の光が浮かんでいるような静かな世界観
- シンプル、上質、未来的
- 技術デモらしさはあるが、開発ツールのようには見せない
- 通常画面では使用色を増やしすぎない
- 来場者が説明なしでも、接続、色変更、音声入力、IMU同期を試せる
- ローカルLLMによる色判断を体験として分かりやすく見せる

## 現在の通常画面

- 背景: 暗い夜空。固定seedの星が低速で視差移動し、一部が白いぼかしを伴って瞬く
- 左上: 低彩度の`SPRIGHT`ロゴ。5回タップで担当者パネルを開く
- 上中央: Bluetooth接続アイコン
  - 切断は灰
  - 処理中は橙
  - readyは緑
- Bluetoothと同期の間: グラデーション時間設定アイコン
  - タップで0～5秒の小型スライダー
- 右上: IMU同期アイコン
  - 有効な姿勢推定が継続している間だけ緑
  - それ以外は灰
- 中央: GLBペンライト
  - 黒鉛色の持ち手
  - 半透明の発光管
  - BloomとComposeのぼかしレイヤーによるネオン光
  - 持ち手正面にyaw方向を判別する非対称マーカー
- 下部: 枠のないマイクアイコン
- マイク上:
  - 上段は文字起こし専用の固定領域
  - 下段は推論ローディング、色スウォッチ、`#RRGGBB`専用の固定領域

## 音声操作の状態

1. 音声入力中
2. 文字認識中
3. 文字起こしを先に表示
4. カラーコード欄でLLM推論ローディング
5. 色スウォッチとHEXコードを表示
6. BLE送信

音声ボタン周囲の円形ローディングは、音声入力中と文字認識中だけ表示します。LLM推論中はカラーコード欄だけにローディングを表示します。BLE送信中、特にグラデーション送信中は音声ボタンを再操作できませんが、音声ボタン周囲のローディングは表示しません。

## 色変更UI

発光部をタップするとモーダルを開きます。

- 現在色と指定色
- グラデーション有効時は両者の間のグラデーション帯
- HSVカラーマップ
- `キャンセル`
- `OK`

グラデーション時間はモーダル内ではなく上部の共通設定を使用し、手動操作と音声操作の両方へ適用します。50 ms間隔、最大100段階でBLE送信します。

## IMU同期UI

- 同期開始時に、ペンライトを水平な机へ置くアニメーションを表示
- 5秒間の緩やかな静止判定
- 成功後だけ3D姿勢を有効化
- 地磁気がないためyawは絶対方位ではなく同期時を0とする相対値

3DモデルはGLB中央を直接回していません。持ち手中央下に`SpresensePivot`親Nodeを置き、GLBを上へオフセットした子Nodeとして配置し、親へIMU Quaternionを適用しています。Spresenseは実物の持ち手内部に搭載されるためです。

## 担当者パネル

通常は非表示です。`SPRIGHT`ロゴを5回タップした場合だけ、次を半透明パネルで表示します。

- BLE状態
- 現在色、RGB、PWM
- 文字認識結果
- ルール / LLM
- 推論時間
- IMU受信Hz
- roll / pitch / yaw
- キャリブレーション統計
- 日本語オフライン音声パック
- 姿勢表示リセット
- 画面固定

## アプリアイコン

adaptive iconとmonochrome iconがあります。現在は次の要素を使用しています。

- 夜空
- 一本の発光ペンライト
- 黒鉛色の持ち手
- yawマーカー
- 低彩度

ホーム画面の小さい表示、円形マスク、角丸四角マスク、単色テーマアイコンのそれぞれで、Sprightだと識別できるか確認してください。

## 実装中に悩んだ点

以下は、見た目と技術制約の折り合いに悩んだ部分です。レビューでは、現状が妥当か、別案がよいかを評価してください。

1. 暗い画面の中で、BLE・同期状態を伝えるための緑と橙が目立ちすぎないか
2. 星空、GLBのBloom、Composeの外側光が競合して、発光が白飛び・過剰になっていないか
3. ペンライトを主役として大きく見せつつ、上下の操作UIと十分な余白を確保できているか
4. 持ち手内部を回転中心にした動きが自然か。発光管が大きく振れたときに画面端へ近づきすぎないか
5. IMU 60 Hzに対して描画フレーム補間を行っているが、展示として滑らかさと追従性のバランスがよいか
6. Android音声認識が途中結果を返さない端末があるため、入力中のライブ文字表示を保証できない。その場合でも「聞いている」ことが十分伝わるか
7. 文字起こし、LLM推論、BLE送信を別状態として見せたい一方、ローディング表現を増やしすぎていないか
8. BLE書き込み成功後だけ画面色を更新するため、操作から見た目の反映までわずかな遅延がある。信頼性と即応感のバランスがよいか
9. 発光部だけをhit test対象にしているが、展示来場者にとってタップ可能箇所が分かりにくくないか
10. アプリアイコンへ夜空、ペンライト、yawマーカーを入れると、小サイズでは情報量が多すぎないか
11. 開発者情報を通常画面から隠した結果、展示担当者がトラブル時に状態を確認しにくくないか

## レビュー観点

以下を必ず評価してください。

- 展示ブースで0.5～2 m離れて見たときの情報階層
- 明るい会場でのコントラストと視認性
- 初見で操作可能箇所を理解できるか
- 状態色とローディングの意味が一貫しているか
- 誤操作や連打を誘発しないか
- タップ領域が十分か
- 文字サイズ、余白、整列、視線誘導
- 星空、発光、3D材質、カメラ、モデル比率の完成度
- IMU回転と回転中心の自然さ
- アニメーションの速度、周期、過剰さ
- ブランドとしての独自性
- UIとアプリアイコンの一貫性
- adaptive icon、round icon、monochrome iconの成立
- アクセシビリティと色覚多様性
- 展示中に処理が失敗した場合の見え方

## 希望する回答形式

日本語で、次の順序で回答してください。

1. 全体評価を5行以内
2. 問題点を`P0 / P1 / P2`の優先度付き表で提示
   - 対象画面または部品
   - 問題
   - 展示で起こる影響
   - 具体的な修正案
3. 通常画面、色変更、音声、IMU同期、担当者パネルを個別レビュー
4. 3Dモデル、回転中心、発光、星空、モーションを個別レビュー
5. アプリアイコンを通常・round・monochromeに分けてレビュー
6. 維持すべき点
7. 最終的な改善案を、可能ならdp、sp、色、透明度、時間、配置比率など具体値で提示

好みだけで断定せず、「展示での理解速度」「視認性」「ブランド一貫性」「技術状態の正確さ」のどれに基づく指摘かを明記してください。判断材料が不足している箇所は、推測で決めず、追加で必要なスクリーンショットや動画を指定してください。
